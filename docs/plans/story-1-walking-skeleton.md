# Story 1 — Walking-skeleton base: schema + health

**Status.** Planned, not yet built. Immutable once implementation lands — see
[BUILD_STATUS.md](../../BUILD_STATUS.md) for live status.

**Source.** [07-epic-map.md](../design/07-epic-map.md) Story 1, elaborated through a
`/grilling` session on 2026-07-13. This file records *why* each choice was made; the
epic map records *what* is required.

---

## Scope

Spring Boot app boots → connects to containerised Postgres → Flyway migration creates
`users` and `time_entries` → `GET /api/health` proves DB connectivity. No auth, no
entry/user endpoints, no client, no JPA `@Entity` classes.

## Decisions made this session (and why)

1. **Health check is DB-readiness, not liveness-only.** `/api/health` runs a live
   `SELECT 1` via `JdbcTemplate`, hand-rolled (not Spring Boot Actuator — the contract
   `/api/health` + `{"status":"ok"}` is small and specific enough that fighting
   Actuator's defaults costs more than writing it directly). Rationale: a liveness-only
   check only proves the process is up, which a running port already tells you. The one
   thing Story 1 actually risks getting wrong is DB wiring (bad `DATABASE_URL`, migration
   ran against the wrong database) — the health endpoint should be the thing that proves
   that wiring, not assume it from a successful boot.

2. **Primary keys are `bigint` identity, not UUID.** `GENERATED BY DEFAULT AS IDENTITY`
   on both `users.id` and `time_entries.id`. UUID's usual justification — stopping ID
   enumeration/IDOR — is already handled here by **INV-2**, enforced server-side from the
   authenticated token, not by ID obscurity. On the read side the model is *shared
   visibility* (any member can already read any entry), so a guessable ID leaks nothing
   not already intentionally shared. UUID would add index/storage cost and a v4-vs-v7
   generation decision to solve a problem (multi-tenancy merge, offline ID generation)
   that is explicitly deferred (ADR-004, ADR "offline & sync" backlog).

3. **Login identifier is `username`, not `email`.** Changed mid-session at the
   developer's direction: this is an internal tool with **no email-driven features at
   all** (no notifications, no email-based reset — ADR-002 already specifies admin
   resets in the DB — no integrations in v1), so an `email` column would be pure
   decoration. This is a domain-model change, not just a schema one — it updated
   [02-domain-model.md](../design/02-domain-model.md) (glossary, `User` entity, INV-5)
   and Story 2's acceptance criteria in the epic map, in the same session, so the docs
   never went out of sync. If email is ever needed later (notifications, self-service
   reset), it's an additive nullable column — a two-way door.

4. **`username varchar(50) NOT NULL`, unique index on `lower(username)`, forced
   lowercase on write.** Case-insensitive uniqueness is enforced at the DB via a
   functional index — not left to application-layer discipline — for the same reason
   INV-5 is stated as a hard invariant: a duplicate `JSmith`/`jsmith` pair, or a
   case-mismatched login lookup, is a real bug class for any login identifier.
   Storing forced-lowercase (rather than as-entered-but-compared-insensitively) was
   chosen for simplicity — there's no display reason to preserve a login handle's
   original casing. Charset/min-length validation (e.g. allowed characters) is
   deliberately **not** a DB constraint — deferred to Story 2's application-layer
   validation, since Story 1 has no user-creation endpoint yet and the DB shouldn't
   carry a format regex under the Floor dial.

5. **`role varchar(20) NOT NULL CHECK (role IN ('member','admin'))`, not a native
   Postgres enum.** Two fixed values, mapped via `@Enumerated(EnumType.STRING)` on a
   Java `Role` enum in Stories 2/3. A native `CREATE TYPE ... AS ENUM` gives the same
   integrity guarantee but is materially harder to evolve (values can't be easily
   removed/reordered, and older Postgres versions can't add a value inside a
   transaction) — ceremony this system doesn't need for two values.

6. **`duration_min integer NOT NULL CHECK (duration_min > 0)`, no upper bound.**
   Directly enforces INV-3 and Story 1's own AC-2. No maximum is imposed — the domain
   model states none, and a hard cap is a business rule better left to future app-level
   validation than baked into the schema now.

7. **`description varchar(500) NOT NULL`.** NOT NULL because, with Projects deferred
   (ADR-007), `description` is the *only* signal in the shared feed of what work was
   done — a null-description row is meaningless. `varchar(500)` bounds runaway input
   while remaining generous. Non-empty/whitespace-only rejection is deferred to Story
   3's create-entry validation (same reasoning as username's charset rule — input
   shape belongs at the app boundary, not a DB CHECK).

8. **Timestamps: `entry_date date`; `created_at`/`updated_at timestamptz NOT NULL
   DEFAULT now()`.** `timestamptz` (not bare `timestamp`) because a team could log in
   from different local times and a timezone-naive column silently drops offset
   information. `updated_at` is advanced by the application (JPA `@PreUpdate` on the
   `TimeEntry` entity in Story 5), not a DB trigger — keeping all business-relevant
   behavior visible in the Java service layer per the layered architecture in
   [04-architecture.md](../design/04-architecture.md), rather than hiding a rule in a
   migration file that a trigger would require. A single Spring app is the only writer
   to this table, so a trigger's main advantage (safety against writers that bypass the
   app layer) doesn't apply here.

9. **No JPA `@Entity` classes in Story 1.** Story 1's own scope boundary says "no
   entry/user endpoints" — by the same logic, there's no reason yet to have the JPA
   mapping layer for entities nothing consumes. Writing `User.java`/`TimeEntry.java` now
   would mean guessing at shape decisions (converters, table/column annotations, Lombok
   or not) that Stories 2 and 3 are better positioned to make against real requirements.
   Story 1 uses `spring-boot-starter-jdbc` (or `data-jpa` with zero entities — either
   auto-configures the `DataSource` and Flyway wiring) + `flyway-core` only.

10. **Testcontainers, not a disposable local DB.** Tests run the real Flyway migration
    against a real ephemeral Postgres engine, so the `CHECK`/unique constraints are
    *actually exercised*, not assumed — and Testcontainers travels with the codebase
    (CI gets the same guarantee a local disposable container can't promise). One
    integration test class for this story: app context loads, `/api/health` → `200
    {"status":"ok"}`, and raw-JDBC assertions that `duration_min > 0` and unique
    `lower(username)` actually reject bad inserts. This is the story's only real test
    surface — there's no API yet, so proving the migration is correct *is* this story's
    integration-layer contribution per [06b](../design/06b-engineering-decisions.md).

11. **Postgres version pinned to `postgres:18-alpine`**, identically in
    `docker-compose.yml` and the Testcontainers image string — not floating `latest`,
    which would let local/CI/Testcontainers silently resolve to different major versions
    on different days. Re-check against the prod PaaS's offered major version at Story 9.

## Deliverables

- `docker-compose.yml` — Postgres 18 service only (datastore-only per ADR-006), env-driven
  credentials matching `DATABASE_URL`.
- `src/main/resources/db/migration/V1__init.sql` — creates `users` and `time_entries` per
  the column spec in [07-epic-map.md](../design/07-epic-map.md) Story 1 AC-2.
- `HealthController` (`GET /api/health`) + a thin DB-ping check via `JdbcTemplate`.
- `.env.example` — `DATABASE_URL`, plus placeholders for `JWT_SECRET`/`JWT_TTL_DAYS`
  (unused until Story 2, but the convention is to declare env surface once — confirm
  with the developer before adding unused-but-declared vars if this feels premature).
- One `@SpringBootTest` + Testcontainers integration test class per point 10 above.

## Explicitly out of scope (do not build)

Auth, entry/user endpoints, any client code, JPA `@Entity` classes, an `email` column,
duration upper bounds, description non-empty validation, username charset validation.
